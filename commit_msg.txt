feat(net): adaptive frame-rate throttle + graceful reconnect under congestion (#XXXX)

Under severe network congestion AetherSDR would drop out after only 5 missed
pings (~5 s) even when the link was merely congested, not dead. The disconnect
then tore down all panadapters and required a full state rebuild. SSDR avoids
this by progressively reducing the FFT/waterfall frame rate to shed UDP load,
and by tolerating more missed pings before declaring the link dead.

────────────────────────────────────────────────────────────────────────────
Part 1 — Adaptive frame-rate throttling (RadioModel)
────────────────────────────────────────────────────────────────────────────
evaluateNetworkQuality() captures the previous NetState before the assignment
and calls applyAdaptiveFrameRate(newState, oldState) only on transitions —
so display pan set fps=N is never re-sent to a pan that is already at the
correct throttled rate.

applyAdaptiveFrameRate() maps state to a (fps, wfMs) pair and returns early
when the fps cap hasn't changed (newCap == oldCap), ensuring VeryGood↔Excellent
churn produces no chatter:

  NetState::Good  → 15 fps / 150 ms wf line_duration
  NetState::Fair  →  8 fps / 250 ms wf line_duration
  NetState::Poor  →  4 fps / 500 ms wf line_duration
  VeryGood/Excellent → signal MainWindow to restore user-configured values

The method iterates m_panadapters.values() directly and sends
"display pan set <panId> fps=N" to every owned pan, not just m_activePanId
(which setPanFps/setWaterfallLineDuration do). Each waterfallId is also
updated if present.

────────────────────────────────────────────────────────────────────────────
Part 2 — Extended ping miss threshold in Poor state (RadioModel)
────────────────────────────────────────────────────────────────────────────
The ping timer fires every 1 s and sends "ping" over the radio control TCP
channel (not OS-level TCP keepalive). When the radio is slow to respond, the
router buffers fill with UDP FFT/audio traffic, crowding out TCP ACKs. Once
the adaptive throttle cuts UDP load, brief TCP stalls are more likely to be
transient congestion than a dead link.

The forced-disconnect threshold is now state-aware:

  Normal (Excellent → Fair): PING_MISS_DISCONNECT = 5  (~5 s) — unchanged
  Poor:                       PING_MISS_DISCONNECT_POOR = 15 (~15 s)

The longer window only applies when NetState is already Poor, so a sudden
hard failure (iptables DROP) still disconnects at 5 s in every other state.

────────────────────────────────────────────────────────────────────────────
Part 3 — fps/wf restore on throttle lift; reconcile and user input handling
         (MainWindow)
────────────────────────────────────────────────────────────────────────────
RadioModel emits adaptiveThrottleChanged(bool active, int fpsCap) on each
engage/lift crossing:

active=true:
  MainWindow sets m_adaptiveThrottleActive = true.
  - schedulePanFpsReconcile() returns early — the radio's echoed fps is
    intentionally below the user setting during congestion; the reconcile
    timer must not fight the throttle.
  - scheduleWaterfallLineDurationReconcile() same guard.
  - The fftFpsChanged lambda (user FPS slider) still calls sw->setFftFps(v)
    so the restore target stays current, but skips the radio sendCommand.
  - The applyWaterfallLineDuration lambda (user WF speed slider/drag) same
    pattern: update sw, skip radio command.
  - The displaySettingsReset handler (user "Reset display settings") skips
    the fps=25 and line_duration=100 radio commands only; all other resets
    (average, weighted_average, color_gain, black_level) still apply.

active=false:
  MainWindow iterates m_panStack->allApplets() and pushes each
  SpectrumWidget's fftFps() and wfLineDuration() back to the radio, restoring
  the user-configured values in one shot.

Cleanup:
  m_adaptiveThrottleActive is cleared in the connectionStateChanged(false)
  handler on disconnect. As a belt-and-suspenders measure, startNetworkMonitor()
  also emits adaptiveThrottleChanged(false, 0) at the start of each new session;
  pans are not yet rebuilt at that point so the restore loop is a no-op.

────────────────────────────────────────────────────────────────────────────
Scope note
────────────────────────────────────────────────────────────────────────────
Only FFT and waterfall UDP streams are throttled. DAX audio, DAX IQ, TX
waveform, meter, and net-CW streams are untouched — a heavily-DAX-loaded
session may still drop out if DAX audio alone saturates the link. A future
follow-on could issue a "stream set ... type=remote_audio_rx compression=alaw"
renegotiation in Poor state, but that is out of scope here.

────────────────────────────────────────────────────────────────────────────
Test cases to exercise before merge
────────────────────────────────────────────────────────────────────────────
Multi-pan:
  Two slices on two panadapters — verify both pans get the throttled fps, not
  just the active one.  Switch active pan during throttle — no re-broadcast.

User slider during congestion (point 3):
  Move FPS slider while in Poor state. Observed: slider value updates in
  SpectrumWidget; radio keeps the throttled fps; no fight. On recovery the
  slider's value is the restored fps.

Induction with tc netem (Linux):
  sudo tc qdisc add dev eth0 root netem loss 5% delay 200ms
  Confirm: network label Poor, lcProtocol logs "adaptive throttle engaged —
  fps cap 4". Clear rule; confirm "adaptive throttle lifted", full fps
  restored with no oscillation.

Hard link death in Poor:
  iptables -A INPUT -p udp --dport 4992 -j DROP (keep TCP alive)
  After ~15 s: forced disconnect, reconnect dialog appears.

Auto-reconnect cleanliness:
  After reconnect m_adaptiveThrottleActive is false (check via network label
  returning to Excellent immediately on fresh link).
