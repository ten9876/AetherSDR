feat(audio): mute-all-slices button + fix stale mute display after reconnect

## What changed

### New: Mute All Slices button (RxApplet) + keyboard shortcut (MainWindow)

A 🔇 button has been added to the right side of the slice-selector tab row in
RxApplet. The tab row is only visible when maxSlices > 1 (multi-slice radios),
so the button appears only where "mute all" is meaningful.

Toggle logic:
  - Any owned slice unmuted → mutes all owned slices
  - All owned slices muted  → unmutes all owned slices

The button emits RxApplet::muteAllToggled(). All toggle logic lives in
MainWindow::onMuteAllSlicesToggle(), which has full RADE context. A
mute_all_slices_toggle keyboard shortcut (category "Audio", no default binding)
calls the same slot via registerAction() / ShortcutManager so it participates in
s_keyboardShortcutsEnabled / shortcutInputCaptured() guards and is editable via
ShortcutDialog.

### RADE interaction (and why the first version was incomplete)

An earlier draft kept the toggle logic in RxApplet and relied on a comment
saying "setAudioMute() no-ops when the value doesn't change, so the RADE slice
is safe." That is true for the MUTE-ALL direction (RADE slice is already muted,
setAudioMute(true) returns early at the existing-value guard, m_radePrevMute is
untouched). It is WRONG for the UNMUTE-ALL direction:

  When every owned slice is muted (RADE slice included), anyUnmuted = false, so
  the loop calls setAudioMute(false) on the RADE slice. That is NOT a no-op —
  m_audioMute changes true→false, the command "slice set N audio_mute=0" goes
  to the radio, RADE's audio gating breaks, and m_radePrevMute is now stale
  (it will restore to whatever value was saved at activateRADE() time, which
  may no longer reflect the user's intent).

Fix: onMuteAllSlicesToggle() skips slices whose sliceId matches m_radeSliceId
in BOTH the scan (anyUnmuted check) and the apply loop, guarded by
#ifdef HAVE_RADE. Moving the logic to MainWindow is the only way to access
m_radeSliceId; RxApplet has no visibility into RADE state and should not.

### Squelch paths — no interaction (confirmed clean)

Auto-squelch is FFT/noise-floor driven (SpectrumWidget::autoSquelchLevelSuggested
→ setSquelch() protocol command). Neither the auto path nor the manual squelch
path ever calls setAudioMute() or sends audio_mute= to the radio. Squelch state
preservation (RxApplet::m_savedSquelchOn) is a completely independent mechanism
that saves/restores squelch_on across mode changes that don't support squelch.
Mute-all does not interact with squelch state in any direction.

### Foreign / unowned slices (confirmed clean, no code change needed)

RadioModel::handleSliceStatus() (RadioModel.cpp:4295-4304) removes a SliceModel
from m_slices and deleteLater()'s it as soon as its client_handle is identified
as belonging to another client. So RadioModel::slices() contains only owned
slices. No additional filtering is needed.

### Bug fix: slice shows as muted after reconnect when radio is not muted

Reported by Robbie KI4TTZ.

**Root cause**

RadioModel::onDisconnected() (RadioModel.cpp:2238) clears panadapters, meters,
and other models on disconnect, but does NOT delete SliceModel objects. They
survive reconnect so the UI does not flicker — intentional for SmartConnect
fast-reconnect, but it creates a stale-state hazard.

The radio does not persist audio_mute across sessions (see MainWindow.cpp
migration block ~line 1265). After any disconnect the radio resets all slice
audio_mute values to 0. When it sends the full slice status on reconnect, it
only includes non-default values; since audio_mute=0 is the default, the key
is omitted entirely. SliceModel::applyStatus() only updates m_audioMute when
kvs.contains("audio_mute"), so a previously muted SliceModel keeps
m_audioMute = true indefinitely — the button shows 🔇 and the model reports
muted, but the radio is actually playing audio.

**Fix** (form suggested by Jeremy KK7GWY in review)

SliceModel::applyStatus() gains an else-if branch:

    } else if (kvs.value("in_use") == "1" && m_audioMute) {
        // Full status w/o audio_mute key → radio reset to default (0)
        // on (re)connect.  Radio does not persist audio_mute
        // (see MainWindow.cpp migration note ~line 1264).
        m_audioMute = false;
        emit audioMuteChanged(false);
    }

in_use=1 is the codebase's canonical marker for a full status message
(RadioModel::handleSliceStatus() already refuses to create a SliceModel from a
status lacking in_use=1). The && m_audioMute guard prevents a spurious
audioMuteChanged(false) on every full-status message for slices that were
already unmuted.

The fix is safe because:
  1. The radio is documented as not persisting audio_mute.
  2. Full-status messages (in_use=1) arrive only on initial slice creation or
     reconnect, not on frequent partial-update messages.
  3. If the radio genuinely sends audio_mute=1 after reconnect, it will be
     present in the KVS and the existing first branch handles it normally.
