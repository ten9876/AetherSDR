feat(audio): mute-all-slices button + fix stale mute display after reconnect

## What changed

### New: Mute All Slices button (RxApplet)

A 🔇 button has been added to the right side of the slice-selector tab row in
RxApplet. The tab row is only visible when maxSlices > 1 (multi-slice radios),
so the button appears only in contexts where "mute all" is meaningful.

Toggle logic:
  - Any slice unmuted → mutes all slices
  - All slices already muted → unmutes all slices

The button calls `SliceModel::setAudioMute()` on every slice owned by the
current RadioModel, so each change sends the correct `slice set N audio_mute=`
command to the radio and emits `audioMuteChanged` to keep all UI in sync.

A matching keyboard shortcut ("Mute All Slices", category "Audio") is
registered in MainWindow as `mute_all_slices_toggle` with no default binding.
Users can assign a key via the shortcut editor.

Files: src/gui/RxApplet.h, src/gui/RxApplet.cpp, src/gui/MainWindow.cpp

### Bug fix: slice shows as muted after reconnect when radio is not muted

Reported by Robbie KI4TTZ: intermittent "slice shows as muted but audio is
playing" after a reconnect.

**Root cause**

`RadioModel::onDisconnected()` clears panadapters, meters, and other models
on disconnect, but does NOT delete `SliceModel` objects. They survive
reconnect so the UI does not flicker. This is intentional for SmartConnect
fast-reconnect, but it creates a stale-state hazard.

The radio does not persist `audio_mute` across sessions (see MainWindow.cpp
migration comment, line ~1265: "The radio does not persist audio_mute, so
restoring it from AppSettings caused Slice A to start muted on every
reconnect"). After any disconnect — even a brief network drop — the radio
resets all slice audio_mute values to 0.

When the reconnected radio sends the full slice status, it only includes
non-default values. If `audio_mute` was 0 (default), the key is omitted from
the status message entirely. The existing `applyStatus()` code only updates
`m_audioMute` when `kvs.contains("audio_mute")`, so a previously muted
SliceModel would keep `m_audioMute = true` indefinitely — the button shows 🔇
and the model reports muted, but the radio is actually playing audio.

**Fix**

In `SliceModel::applyStatus()`, the `audio_mute` block is extended with an
`else if` branch that fires when:
  - `audio_mute` is absent from the KVS, AND
  - `in_use=1` is present (marking this as a full status, not a partial update)

In that case the code resets `m_audioMute` to `false` and emits
`audioMuteChanged(false)` if needed. This is safe because:
  1. The radio is documented as not persisting audio_mute.
  2. Full-status messages (in_use=1) arrive only on initial slice creation or
     on reconnect — not on the frequent partial-update messages that carry only
     the key(s) that changed.
  3. If the radio genuinely wants the slice muted after reconnect it will
     include audio_mute=1 in the full status, which the existing branch handles.

File: src/models/SliceModel.cpp
